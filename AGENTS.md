# Veloce Project Rules & Agent Guidelines

## 1. 対象環境と技術スタック
- **動作ターゲット**: Windows 8.1 をサポート範囲に含める。
  - *注記: これは現作者のローカル環境の都合によるものです。フォークした他の開発者はこの制約を守らなくても構いません。*
- **技術制約 (WebView2 v109)**: Windows 8.1環境で動作させるため、Chromium 109 互換の JavaScript / CSS を出力すること（v110以降の新しい配列メソッドやCSSネストなどは使用厳禁）。
- **技術スタック**: Tauri (Rust) + SQLite + Vanilla JS のみを使用する。React等の仮想DOMフレームワークや肥大化するライブラリの導入は禁止。
- **処理対象となる画像ファイル**: NovelAIユーザをメインターゲットとし、処理する画像形式はpngやwebpがメインとなる。パフォーマンスチューニングもそれらがメインとなる。jpgなどそれ以外の画像形式は、「WebViewが対応している」「一般的な画像形式である」ということで対応しているだけなので、そちらに注することは禁止。

## 2. UI / デザインガイドライン
- **絵文字の厳禁**: Windows 8.1 で正常に表示できないため、UI上（およびAIのチャット出力等）への絵文字（⭐など）の使用は厳禁とする。
- **アイコンの使用**: アイコンなどが必要な場合は、絵文字ではなく SVG アイコンを使用し、他のUIコンポーネントとテイストを統一すること。

## 3. パフォーマンスとアーキテクチャ設計
- **極限の軽量化**: メモリ割り当てやガベージコレクション（GC）の発生を最小限に抑えること。
- **DOM操作の最適化**: `DocumentFragment` やイベント委譲（Event Delegation）を活用し、O(1)の差分更新を徹底すること。再計算（Reflow/Repaint）を誘発する非効率なDOM操作は避ける。
- Veloceという名前に恥じないパフォーマンスを維持すること。とくにサムネイル作成や表示については、画像ビューアの根幹となる部分である。ミリ秒以下の単位での処理にすること。

## 4. テストと開発フロー
- **【重要・厳守】** 実装コード（機能追加・修正）を出力する際は、**必ず同一の回答（または同一のステップ）内でペアとなるテストコードも出力すること**。「テストは後で実装します」「〜と同様にテストを追加してください」等の省略・先送り（サボり）は固く禁ずる。
- Rust、JavaScriptを修正したときはテストコードを記載し、既存のコードも含めて実行すること。
- 既存のコードの修正や機能追加の際は、必要に応じて既存のテストコードも修正すること。
- 新規のコードを記載する際は既存のテストコードを参考に、命名規則やコードスタイルを統一すること。
- 不具合として指摘された内容についてはテストに追加し、修正後は必ずテストをクリアすること。
- 新規の機能追加の際は、必ずテストコードも新規に記載すること。
- 編集用に作成したスクリプト（*.cjsなど）は、反映が終わったら削除すること。
- 自己のテスト用に作成したデータファイルは、確認が終わったら削除すること。

## 5. リポジトリの運用方針
- 本プロジェクトはAI生成コードを主体として開発を行っているため、外部からの Pull Request (PR) は受け付けていない。
